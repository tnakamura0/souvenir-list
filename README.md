## 1. サービス概要

旅行や出張でお土産を買う人が、買い忘れの不安から解放され、最後まで安心して旅行を楽しめるサービスです。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

私は旅行に行った際に、家族や職場など複数の人へお土産を購入してきました。

旅行中は観光や移動など予定が多く、お土産を買うタイミングも限られています。「あの人のお土産は買ったかな」「まだ買っていない人はいないかな」と何度も頭の中で確認しながら行動していました。

旅行には帰る時間というタイムリミットがあります。もし買い忘れに気付いたとしても、帰りの交通機関まで時間がなかったりして思い通りに購入できないこともあります。

せっかくの旅行を楽しみたい気持ちがある一方で、「誰かのお土産を買い忘れてしまうかもしれない」という不安を抱えながら過ごしていたことが、このアイデアのきっかけです。


---

### 2-2. なぜそれが「気になった」のか

私は、相手への気遣いを大切にしたいと考えています。

お土産は必ず買わなければならないものではありませんが、「旅行へ行ったらこの人にも買って帰ろう」と思った相手には、できるだけ忘れずに渡したいという気持ちがあります。

また、旅行中は普段以上に移動や観光で体力・気力を使うため、頭の中だけで購入状況を管理することに負担を感じていました。

「旅行を楽しみたいのに、常に買い忘れを気にしてしまう」という状態を改善できれば、旅行そのものをもっと安心して楽しめるのではないかと考え、このアプリを作ろうと思いました。

---

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

旅行中、お土産を買う相手が複数いると、「誰に買ったか」「まだ買っていない人はいないか」が分からなくなり、買い忘れてしまうかもしれないという不安を感じる。

---

### 3-2. 本当に解決したい課題は何か

- なぜ？
	- 買い忘れると、帰宅後には取り返すことができず、「あの人にだけ買えなかった」という後悔が残るから
- なぜ？
	- お土産を渡したいと思っていた相手への気持ちを十分に形にできず、申し訳なさや残念な気持ちになるから
- なぜ？
	- その後悔を避けようとして、旅行中も「買い忘れていないか」を何度も確認し続ける必要があり、旅行を心から楽しめなくなるから

本当に解決したい課題：お土産の買い忘れを気にし続けることなく、安心して旅行そのものを楽しめる状態を実現したい


---

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

仕事やプライベートで旅行へ行く機会があり、家族や職場など複数人へお土産を買う30代。相手への気遣いを大切にしており、「この人にも買って帰りたい」と思った相手へのお土産を買い忘れたくないため、旅行中も購入状況を何度も確認してしまう人。

---

### 4-2. 自分とユーザーの距離

私自身、旅行へ行くたびに「誰の分を買ったか」「まだ買っていない人はいないか」を旅行中に何度も確認してしまい、買い忘れへの不安を感じていました。

### 4-3. 実際に使われる可能性について(需要・存在確認)

#### まず最初に使ってくれそうな人
- 家族・職場・友人など5人以上へお土産を購入する人
- 過去に買い忘れた経験がある人、または旅行中に「買い忘れていないかな」と不安になった経験がある人
#### どんな場面でこのアプリを思い出すか
- 旅行前に「今回は誰にお土産を買おう」と考えたとき
- お土産売り場で購入する相手を確認したいとき

#### 現在代わりに使われているもの
- 記憶
- スマートフォンのメモアプリやチェックリスト

#### それを置き換えてまで使う理由

現在は記憶やメモアプリで管理している人が多いと考えられます。しかし、メモアプリは自由度が高い反面、旅行ごとに渡す相手を入力する手間があり、購入状況も一覧で把握しにくいという課題があります。

本サービスでは、「旅行」と「渡す相手」を軸にお土産を管理できます。誰に対して購入したか、まだ購入していない相手は誰かを一目で確認できるため、旅行中の買い忘れへの不安を減らし、安心して旅行を楽しめると考えています。

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ(導入・継続の流れ)

旅行へ行くことが決まり、「今回は誰にお土産を買おう」と考え始めたタイミングで本サービスの利用を始めます。

利用開始後は、旅行を登録し、お土産を渡したい相手を登録します。旅行中は、お土産を購入した相手にチェックを付けるだけで購入状況を記録でき、帰宅前には買い忘れがないかを一覧で確認できます。

入力項目を最小限にすることで、旅行前でも負担なく利用を始められるようにします。

#### 利用を続ける理由

- 一度登録した「お土産を渡す相手」は次回以降の旅行でも再利用できるため、毎回入力し直す必要がない

- 「持ち物の確認」「交通機関・ホテルの確認」と同じように、「お土産を渡す相手を確認する」ことを旅行準備の習慣として取り入れられる

- 旅行の最後に買い忘れがないことを確認できるため、不安を残さず安心して旅行を終えられる

## 5. 既存サービス・競合調査

  - 家族や友人と共有して共同編集ができる
#### スマートフォン標準のメモアプリ
- **どんなことができるか**
  - 自由にメモやチェックリストを作成できる
  - 旅行ごとにメモを作成して管理できる

---

### 5-2. それでも自分が作りたい理由

既存サービスを調査した結果、旅行全体を管理するサービスや、自由にメモを残せるサービスは存在していました。
しかし、私が解決したかったのは「旅行中のお土産管理」でした。
旅行管理サービスは旅程や持ち物など旅行全体を管理することを目的としており、「誰にお土産を買ったか」「まだ買っていない相手は誰か」を確認するには向いていませんでした。
また、メモアプリは自由に記録できる一方で、旅行ごとに渡す相手を毎回入力する必要があり、購入状況も一覧で把握しづらいと感じました。

私が欲しかったのは、「買い忘れを防ぐためのメモ」ではなく、旅行中に「もう買い忘れはない」と安心できる仕組みです。
そのような体験を提供するサービスが見当たらなかったため、自分で作りたいと考えました。

### 5-3. 差別化を「体験価値」で表現する

旅行や出張で複数人へお土産を買う人が、「誰に買ったか」「まだ買っていない人はいないか」という不安から解放され、安心して旅行を楽しめる体験を提供します。

---

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化

- 行動の変化：記憶に頼らず、お土産の購入状況を管理できるようになる
- 気持ちの変化：お土産の管理を気にする時間よりも、旅行を楽しむ時間を大切にできるようになる
- 考え方の変化：お土産は「覚えておくもの」から、「アプリで管理するもの」へ変わる

### 6-2. 価値を一文で表す

このサービスは、旅行や出張で複数人へお土産を買う人が、渡す相手と購入状況をまとめて管理し、買い忘れの不安なく旅行を楽しめるようになるサービスです。

---

## 7. このアプリで実現すること

### MVPで作る機能
MVPでは「旅行中の買い忘れへの不安を減らし、安心して旅行を楽しめる」という体験を実現するため、旅行・お土産を渡す相手の管理と旅行前から旅行中まで負担なく利用できる機能を実装します。
- Googleアカウントで素早く利用を開始できる
- 旅行ごとに目的地や日程を登録し、お土産を管理できる
- お土産を渡したい相手を登録、管理できる
- 「家族」「職場」などのタグで相手を分類できる
- タグで相手を絞り込み、まとめて旅行へ追加できる
- 購入した相手をワンタップで購入済みにできる
- 購入済み・未購入の相手を一覧や進捗バーで確認できる

### 本リリースで作る機能
次回の旅行をより便利にする機能やユーザー体験が向上する機能を追加します。
- 相手ごとに過去のお土産履歴を確認し、次回のお土産選びの参考にできる
- スマートフォンのホーム画面から素早く起動し、旅行中でもすぐに購入状況を確認できる

## 8. このアプリの懸念点とその対策

### 懸念点①

- 何が問題になりそうか：旅行や出張の時しか利用しないため、利用頻度が低く、継続して使われない可能性がある
- なぜそう思うか：本サービスは日常的に使うアプリではなく、旅行という限られたタイミングで利用するサービスだから
- そのために考えている対策：一度登録した相手を次回以降も使いまわせるようにして、旅行のたびに入力し直す手間を減らす。またPWA対応によりホーム画面から素早く起動できるようにして、「旅行前の準備」の流れの中で自然に利用してもらえることを目指す

### 懸念点②

- 何が問題になりそうか：旅行前に相手を登録すること自体が面倒で利用されない可能性がある
- そのために考えている対策：タグ、相手の再利用、一括追加などで入力を最小限にする

## 9. 今後の展開・発展の方向性

### 観点

- ユーザーはなぜ使い続ける？
  - 一度登録した相手や旅行の情報を再利用できるため、旅行や出張のたびに一から準備する手間を減らせる

- 続けることで何が変わる？
  - 相手や購入履歴が蓄積されることで、旅行前の準備をより短時間で行えるようになる

- データが溜まると何ができる？
  - 相手ごとの購入履歴を活用し、以前購入したお土産の確認や、重複購入を避けるためのサポート機能へ発展できる

- 他の人と関わる余地はある？
  - 現時点では個人利用を想定しているが、将来的には家族旅行などで購入状況を共有しながら管理できる仕組みへ発展できる

### 今考えている発展案

- 機能追加の方向性
  - 相手ごとの購入履歴を確認し、次回のお土産選びに活用できる機能
  - 旅行終了前に未購入の相手を通知し、買い忘れをさらに防ぐ機能

- UI・体験の改善
  - よくお土産を渡す相手を優先表示し、旅行前の準備をよりスムーズにする
  - スマートフォンで片手でも操作しやすいレイアウトへ改善する

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails 8.1.x
- 開発言語：Ruby 3.4.x
- フロントエンド：Hotwire（Turbo + Stimulus）
- CSSフレームワーク：Tailwind CSS v4
- DB：PostgreSQL（Neon）
- 認証：OmniAuth（omniauth-google-oauth2）
- テスト：RSpec
- 開発環境：Docker
- CI/CD：GitHub Actions
- デプロイ先：Render

---

### 10-2. 技術選定の比較検討

#### 採用予定

| 分類             | 技術                        | 理由                                                              |
| --------------- | ------------------------- | ---------------------------------------------------------------- |
| フレームワーク    | Ruby on Rails 8.1.x       | サポート期間が長く、長期的に安心して開発できるため。また将来的にPWA対応を検討しており、Rails 8系の標準機能で対応しやすいと判断したため。 |
| 開発言語 | Ruby 3.4.x                | Rails 8.1系の標準構成に合わせ、安定した開発環境を構築するため。                                       |
| フロントエンド    | Hotwire（Turbo + Stimulus） | RailsだけでSPAライクな操作性を実現でき、JavaScriptを最小限に抑えるため。                            |
| CSSフレームワーク | Tailwind CSS v4    | メイン機能であるお土産リストの確認はスマートフォンでの利用が考えられ、画面サイズに応じたUIを効率よく構築できるため。 |
| DB             | PostgreSQL（Neon）           |   PostgreSQL互換であり、無料で利用できるクラウドDBとしてRailsとの相性も良いため。                    |
| 認証            | OmniAuth（omniauth-google-oauth2）   | 旅行中でもログインの手間を減らし、すぐに利用できるようにするため。                                   |
| テスト          | RSpec                     | Railsのデファクトスタンダードとなっているテストフレームワークであり、コード品質を担保するため。           |
| 開発環境        | Docker                    | 開発環境の差異をなくし、再現性の高い環境で開発できるようにするため。                                    |
| CI/CD          | GitHub Actions            | テストやデプロイを自動化し、品質を維持しながら安全にリリースするため。                                  |
| デプロイ        | Render                    | Railsアプリを素早く公開でき、CI/CDとの連携もしやすいため。                                         |

#### 採用しない技術とその理由
| 分類      | 技術  | 理由                                              |
| ------- | ----- | --------------------------------------------------- |
| フロントエンド | React | SPAの開発が目的ではないため、開発速度を優先し、Railsの機能を活かせるHotwireを採用した。 |
| デプロイ | AWS | 個人開発ではインフラ構築・運用の負担が大きくなるため、まずはRenderで素早く公開し、必要に応じて移行を検討する。  |

#### 今回チャレンジしたい点
- 本リリースではPWA対応を実装し、旅行中でも素早く利用できる体験を実現する

#### 不安な点
- スマートフォンで片手操作しやすいUI設計
